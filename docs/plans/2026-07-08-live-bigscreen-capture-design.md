# 直播蓝屏自动截图第一版设计方案

## 背景

直播业务每天开播后，需要在每个整点保存京东直播实时大屏的数据截图，用于后续填写和复盘大屏数据。

最初需求看起来只是“每小时截一张蓝屏大屏图”。根据真实业务场景补充后，实际需要截图的不止总览大屏，还包括流量曲线、渠道占比、用户画像、挂袋商品、订单 Top10 和 GMV Top10 等页面状态。

第一版目标不是做复杂配置平台，而是把业务现在人工每小时切页面、切标签、截图的动作自动化。

## 目标

在现有直播运营工具统一页面中新增一个标签页：

```text
蓝屏自动截图
```

业务用户只需要粘贴蓝屏页面链接，选择今天需要截图的整点，程序在每个整点自动打开对应直播大屏，按固定截图清单依次切换页面状态并保存截图。

第一版默认截取所有已确认的并集项，不让用户配置“截哪些”。后续稳定后再做可配置截图项。

## 非目标

第一版不做：

- 不管理多个直播间。
- 不要求填写直播间名称。
- 不自动从直播列表寻找“直播中”直播间。
- 不自动判断直播是否已经结束。
- 不做跨天复杂排班。
- 不做服务端集中存储。
- 不做截图项自定义配置。
- 不解析图片里的数值，不生成结构化数据报表。

## 用户入口

入口仍沿用现有本地 Web 工具：

```text
http://127.0.0.1:8080
```

页面顶部新增标签页：

```text
蓝屏自动截图
```

页面只服务当前账号对应的一个直播间。业务上每个直播间由各自直播账号操作，因此不需要在一个页面内维护多行直播间。

## 页面表单

页面包含以下区域：

| 区域 | 字段或操作 | 说明 |
| --- | --- | --- |
| 蓝屏链接 | 蓝屏页面链接 | 用户粘贴完整链接，例如 `https://jlive.jd.com/bigScreen?id=46794566` |
| 整点计划 | 整点按钮 | 展示今天可选整点，例如 `12:00` 到 `23:00` |
| 整点计划 | 一键添加今天剩余整点 | 从当前时间后的下一个整点开始生成 |
| 整点计划 | 开始整点、结束整点、生成整点 | 用户可按直播开播到结束时间批量生成 |
| 任务控制 | 开始自动截图 | 校验链接和整点计划后启动 |
| 任务控制 | 停止任务 | 停止后续整点，不删除已生成截图 |
| 临时操作 | 立即截图一次 | 用于补截或测试位置 |
| 运行状态 | 当前状态、已截图数量、失败数量、下次截图时间 | 前端轮询后端状态 |
| 结果 | 下载全部截图 ZIP | 任务执行后可下载当前任务所有截图 |

## 链接规则

用户只填写完整蓝屏链接：

```text
https://jlive.jd.com/bigScreen?id=46794566
```

程序从 URL 参数中解析直播间 ID：

```text
46794566
```

校验规则：

- 链接不能为空。
- 链接域名必须是 `jlive.jd.com`。
- 路径应包含 `/bigScreen`。
- 查询参数必须包含 `id`。

## 整点规则

程序严格按用户选择的整点执行，不按“启动后每 60 分钟”计算。

示例：

当前时间为 `18:35`，用户选择：

```text
19:00, 20:00, 21:00
```

程序只在今天的 `19:00`、`20:00`、`21:00` 执行截图。

如果用户选择了已经过去的整点，例如当前时间是 `18:35` 但选了 `18:00`，该整点标记为：

```text
已错过
```

默认不自动补截。用户需要补截时点击 `立即截图一次`。

## 截图清单

第一版每个整点默认截取以下 15 项。顺序固定，便于后续人工核对和排查失败点。

| 序号 | 截图项 | 页面位置 | 操作口径 |
| --- | --- | --- | --- |
| 01 | 概览总览 | `概览` | 直接截整页 |
| 02 | 渠道流量饼状图_在线 | `概览` -> 左侧 `直播间` 卡片 -> `在线` 标签 | 截左侧直播间卡片，包含在线趋势和在线占比 |
| 03 | 渠道成交饼状图_成交 | `概览` -> 左侧 `直播间` 卡片 -> `成交` 标签 | 截左侧直播间卡片，包含成交趋势和成交占比 |
| 04 | 挂袋数据 | `概览` 中间核心指标上方 `全部商品` 下拉 -> 选择 `挂袋商品` | 切换后截整页 |
| 05 | 在线人数曲线 | `流量` -> `综合趋势` -> `在线人数` | 截流量页主图区域或整页 |
| 06 | 访问人数曲线 | `流量` -> `综合趋势` -> `访问人数` | 截流量页主图区域或整页 |
| 07 | 人均停留时长曲线 | `流量` -> `综合趋势` -> `人均停留时长` | 截流量页主图区域或整页 |
| 08 | 成交人数曲线 | `流量` -> `综合趋势` -> `成交人数` | 截流量页主图区域或整页 |
| 09 | 成交金额曲线 | `流量` -> `综合趋势` -> `成交金额` | 截流量页主图区域或整页 |
| 10 | 成交单量曲线 | `流量` -> `综合趋势` -> `成交单量` | 截流量页主图区域或整页 |
| 11 | 曝光点击率曲线 | `流量` -> `综合趋势` -> `直播曝光点击率` | 截流量页主图区域或整页 |
| 12 | 用户画像_访问用户 | `概览` -> 右侧 `用户画像` -> 下拉选择 `访问用户` | 截用户画像卡片或整页 |
| 13 | 用户画像_成交用户 | `概览` -> 右侧 `用户画像` -> 下拉选择 `成交用户` | 截用户画像卡片或整页 |
| 14 | 订单Top10 | `商品` -> `商品分析` -> 点击表头 `成交件数` 降序 | 截商品页表格 |
| 15 | GMVTop10 | `商品` -> `商品分析` -> 点击表头 `成交金额` 降序 | 截商品页表格 |

说明：

- `直播渠道流量饼状图` 的正确位置是 `概览` 左侧 `直播间` 卡片的 `在线` 标签，不是 `流量` 页的 `流量渠道`。
- `直播渠道成交饼状图` 的正确位置是 `概览` 左侧 `直播间` 卡片的 `成交` 标签。
- `挂袋数据` 的正确入口是 `概览` 中间核心指标上方的商品范围下拉框，将 `全部商品` 切成 `挂袋商品` 后截图整页。
- `直播间订单Top10` 使用商品页表格按 `成交件数` 降序。
- `直播间GMVTop10` 使用商品页表格按 `成交金额` 降序。

## 文件命名

每张截图使用固定命名格式：

```text
蓝屏数据截图_{id}__{YYYYMMDD}_{HHmmss}_{序号}_{截图项}.png
```

示例：

```text
蓝屏数据截图_46794566__20260708_190000_01_概览总览.png
蓝屏数据截图_46794566__20260708_190000_02_渠道流量饼状图_在线.png
蓝屏数据截图_46794566__20260708_190000_03_渠道成交饼状图_成交.png
蓝屏数据截图_46794566__20260708_190000_14_订单Top10.png
蓝屏数据截图_46794566__20260708_190000_15_GMVTop10.png
```

ZIP 包按任务打包：

```text
蓝屏数据截图_46794566__20260708.zip
```

ZIP 内建议按整点分目录：

```text
20260708_190000/
  蓝屏数据截图_46794566__20260708_190000_01_概览总览.png
  ...
20260708_200000/
  蓝屏数据截图_46794566__20260708_200000_01_概览总览.png
  ...
截图清单.xlsx
```

## 截图清单文件

每个任务生成一个 `截图清单.xlsx`，便于核对成功和失败项。

字段：

| 字段 | 说明 |
| --- | --- |
| 计划整点 | 用户选择的整点，例如 `19:00` |
| 实际执行时间 | 程序开始执行该截图项的时间 |
| 直播间ID | 从链接解析出的 ID |
| 序号 | 截图项序号 |
| 截图项 | 截图项名称 |
| 文件名 | 生成的 PNG 文件名 |
| 状态 | `成功` / `失败` / `已错过` / `已停止` |
| 失败原因 | 失败时记录异常信息 |

## 技术架构

沿用现有直播运营工具的分层方式：

```text
live/
  live-web/
    app.py
    templates/index.html
    tests/

  live-bigscreen-capture/
    bigscreen_capture/
      __init__.py
      url_parser.py
      schedule.py
      manifest.py
      browser.py
      capture_steps.py
      archive_writer.py
      service.py
    tests/
```

职责划分：

| 模块 | 职责 |
| --- | --- |
| `live-web` | 页面、API、任务状态、启动停止、下载 |
| `live-bigscreen-capture` | 链接解析、整点计划、截图步骤编排、文件命名、清单、ZIP |
| `live-sku-price-audit/utils/browser_manager.py` | 复用现有 Playwright 登录态、浏览器启动和登录检查能力 |

不把截图业务逻辑写进 `live-web/app.py`。`live-web` 只调用 `live-bigscreen-capture` 提供的服务函数。

## 后端 API 草案

新增接口挂在 `live-web`：

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/bigscreen-capture/preview` | `POST` | 校验链接，返回解析出的直播间 ID 和可选整点 |
| `/api/bigscreen-capture/start` | `POST` | 启动整点截图任务 |
| `/api/bigscreen-capture/status` | `GET` | 查询任务状态 |
| `/api/bigscreen-capture/stop` | `POST` | 停止后续截图 |
| `/api/bigscreen-capture/capture-now` | `POST` | 立即按 15 项清单截图一次 |
| `/api/bigscreen-capture/download` | `GET` | 下载 ZIP |

任务状态字段：

```text
running
stopping
room_id
planned_slots
current_slot
current_step
total_steps
completed_steps
success_count
fail_count
missed_count
next_run_at
result_zip
logs
error
need_login
```

## 浏览器和登录态

第一版复用现有 Playwright 浏览器能力：

- 使用 `live-sku-price-audit` 中已有的 `jd_auth.json` 登录态文件。
- 登录态有效时直接打开蓝屏链接。
- 登录态失效时，沿用现有 Web 工具中的登录提示流程：
  - 后端标记 `need_login=true`。
  - 前端弹出登录提示。
  - 用户在打开的京东登录页完成登录。
  - 用户点击 `我已登录，继续`。
  - 后端重新检查登录态并继续任务。

## 截图步骤状态机

每个整点执行流程：

```text
进入整点
  -> 打开蓝屏链接
  -> 等页面稳定
  -> 执行 01 概览总览
  -> 执行 02 概览/直播间/在线
  -> 执行 03 概览/直播间/成交
  -> 执行 04 概览/挂袋商品
  -> 执行 05-11 流量/综合趋势各指标
  -> 执行 12-13 概览/用户画像下拉
  -> 执行 14-15 商品页表格排序
  -> 写入截图清单
  -> 等待下一个整点
```

单个截图项失败时：

- 记录该项失败。
- 继续执行后续截图项。
- 不因单项失败中断整点任务，除非浏览器或登录态整体不可用。

## 异常处理

| 场景 | 处理 |
| --- | --- |
| 链接为空 | 页面提示填写蓝屏页面链接 |
| 链接缺少 `id` | 页面提示无法识别直播间 ID |
| 登录态失效 | 触发现有登录确认流程 |
| 指定整点已过 | 标记 `已错过` |
| 页面加载失败 | 当前截图项记失败，写入失败原因 |
| 某个按钮找不到 | 当前截图项记失败，继续后续项 |
| 用户点击停止 | 停止后续整点，当前正在执行的截图项尽快收尾 |
| ZIP 不存在 | 下载接口返回明确错误 |

## 验收标准

使用真实链接：

```text
https://jlive.jd.com/bigScreen?id=46794566
```

至少完成一次 `立即截图一次` 验证：

- 能解析直播间 ID。
- 能触发或复用登录态。
- 能生成 15 张截图。
- 15 张截图文件名符合规则。
- `02_渠道流量饼状图_在线` 是概览左侧直播间在线标签。
- `03_渠道成交饼状图_成交` 是概览左侧直播间成交标签。
- `04_挂袋数据` 是概览中间下拉切到挂袋商品后的整页。
- `14_订单Top10` 是商品页按成交件数降序。
- `15_GMVTop10` 是商品页按成交金额降序。
- 能生成 `截图清单.xlsx`。
- 能下载 ZIP。

## 风险和后续优化

主要风险：

- 京东直播大屏 DOM 或按钮文案变化导致截图步骤失效。
- 大屏自动刷新时，截图瞬间数据可能变化。
- 某些账号或直播间没有部分模块，导致个别截图项失败。
- 页面横向或纵向滚动位置会影响截图内容。

第一版应尽量采用“可见文本 + 稳定区域”的方式定位，并在失败时保留日志，不静默跳过。

后续优化：

- 支持用户配置截图项。
- 支持只截卡片区域，不截整页。
- 支持对失败项单独补截。
- 支持截图后预览。
- 支持多账号或多直播间批量任务。
